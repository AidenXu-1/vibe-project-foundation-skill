# Agent 协作规则

> 这是所有 AI Agent 共用的工作规则。`AGENTS.md` 和 `CLAUDE.md` 只做入口,不要重复写长规则。

## 读取策略

为节省上下文,先读必要文件,不要一上来全量读取整个仓库。

| 场景 | 必读 | 按需读 |
|------|------|--------|
| 每次开始任务 | `docs/spec.md`、`docs/progress.md` 最新记录 | `docs/overview.md`、`docs/roadmap.md` |
| 技术选择 / 新依赖 | `docs/spec.md`、`docs/decisions/` | `docs/conventions.md` |
| 接续中断工作 | `docs/handoff.md`、`docs/progress.md` | 相关代码和决策记录 |
| 大改方向 | `docs/spec.md`、`docs/roadmap.md`、`docs/decisions/` | `docs/overview.md` |
| 写代码 | `docs/spec.md`、`docs/conventions.md` | 相关代码、测试、已有相似实现 |

`docs/spec.md` 是当前唯一开发准绳。其他文档如果和 Spec 冲突,以 Spec 为准,并提醒用户同步修正。

## 开发节奏

1. **Foundation**:只搭地基,不选技术栈,不写业务代码。
2. **Spec v0**:先写清楚问题、用户、场景、边界和成功标准。
3. **Risk Check**:先找最大风险,可能是需求、UI、数据、权限、成本或技术可行性。
4. **Spec v1**:冻结 MVP 合同,写清功能范围、技术路线、验收标准和不做什么。
5. **Kickoff**:用户确认后再初始化 `app/`,补齐命令、环境变量示例和首次可运行验证。
6. **Build Loop**:一次只做一个功能切片,计划 → 实现 → 验证 → 更新进度 → commit。

## 阶段门槛

- **进入 Spec v1 前**:必须说清目标用户、核心场景、MVP 范围、不做什么、验收标准和主要风险。
- **进入 Kickoff 前**:必须完成风险检查,把结论写入 `docs/spec.md` 或 `docs/decisions/`,并确认是否需要 UI 原型、数据验证或技术小实验。
- **初始化 `app/` 前**:必须得到用户确认的技术路线,重要选择写入 `docs/decisions/`;同时定下 `app/` 内部布局——源码、测试、配置各放哪——写入 `docs/conventions.md`,此后所有代码和测试按此归位,不要随手乱放。
- **进入 Build Loop 前**:项目必须能以空骨架启动或通过最小验证。

## 改动前检查

- 写代码或改文件前先看当前变更状态,确认有没有用户或其他 Agent 留下的未提交改动。
- 不要覆盖、删除或回滚不属于本次任务的改动;如果这些改动影响当前任务,先向用户说明再继续。

## 改动分级

- **A 级:高风险**。删除数据、改权限/认证/支付/密钥、发送消息、调用付费 API、改部署、跨多文件大重构。先写方案和回退方式,等用户明确确认。
- **B 级:用户可感知**。功能、UI、文案、单模块业务逻辑。动手前简述意图,做完验证。
- **C 级:局部可逆**。文档、注释、typo、测试、小范围整理。可直接做,但仍要说明结果。

## 安全边界

- 未经用户确认,不提交 `.env`、token、密码、私有链接、手机号、邮箱、身份证等真实敏感信息;必须出现时优先用示例、脱敏或环境变量。
- 新增依赖前先说明用途、替代方案和风险。
- 不理解的代码先问,不要直接删。
- 大改先更新 `docs/spec.md`;如果 Spec 与其他文档冲突,以 Spec 为准并提示同步修正。
- 涉及删除、付款、授权、发送消息、发布内容、生产配置和外部付费 API 时,必须先让用户明确确认。
- 实验脚本、调试输出、临时大文件放 `scratch/`(用完即弃,不提交);**正式测试是代码,属于 `app/`**,按 `docs/conventions.md` 定的布局归位,不要丢进 `scratch/`。

## 完成标准

完成必须同时满足:

- 改动符合 `docs/spec.md`。
- 已验证,并说明怎么验证。
- 更新 `docs/progress.md`。
- 一个完整功能切片完成后应提交;如果暂不提交,说明原因。
- 需要交接时更新 `docs/handoff.md`。

## 变更控制

长期维护产品的大改不要直接进代码。先在回复或 `docs/spec.md` 里说明:

- 改什么
- 为什么改
- 影响哪些页面、数据、接口或文件
- 如何验证
- 如何回退

用户确认后,再更新 Spec 和代码。

## UI / 设计可视化确认

凡涉及 UI、交互、视觉呈现、页面布局、设计稿、用户体验路径的节点,设计产出不能只交文字说明、ASCII 线框、Markdown 表格或抽象结论;必须提供用户可直接判断的可视化预览。

- 优先使用 OpenDesign 等专用设计工具生成可编辑设计产物或 artifact。
- 如果当前会话未热加载 OpenDesign MCP、OpenDesign 没有 active project、权限不足或工具连接失败,必须明确说明失败原因,并用本地 HTML + PNG 截图、Figma、可打开图片预览等方式兜底。
- OpenDesign 接入时,先确认本机 OpenDesign App 是否运行,再确认 daemon 健康状态;不要假设默认端口一定是 7456,应从 OpenDesign 日志或本机监听端口确认实际 daemon URL。
- 若 OpenDesign artifact 写入提示没有 active project,要求用户在 OpenDesign 内创建或点进项目,或使用兜底方案交付,不能卡住节点。
- 未安装 / 未运行 OpenDesign、权限不足、连接失败或 MCP 未热加载时,主动询问用户是否需要帮忙安装 / 启动 / 授权 / 注册 MCP / 重载或新开会话。
- 用户不想处理 OpenDesign 或不愿意重载 / 新开会话时,不得卡住,应按用户偏好直接选择本地 HTML + PNG、Figma 或可打开图片预览。
- 设计回报必须同时包含设计说明文档路径和可视化预览路径;若使用兜底方案,写清 OpenDesign 当前状态和后续恢复条件。
- 用户确认设计视觉或交互方向前,不得进入正式实现。用户只反馈功能方向 OK 但 UI 未确认时,只能推进功能可行性或技术评估。
